The two simplest sorts — how they work, and why they still earn their keep.
Author

Vikram Dixit

Published

July 10, 2026

Sorting is one of the most basic approaches to solving problems in human thought. Be it prioritizing which tasks to tackle first or who to pay first; ranking in competitions, exams, and any other comparative decisions — we are sorting. It is a fundamental problem in computer science too: organizing records by date, tabulating results into a sequence of some kind. In this post we look at two of the simplest sort algorithms — selection sort and insertion sort. They are instructive, and although less efficient than other algorithms on large, randomly ordered data, they remain widely used at small sizes and on streaming data. The goal here is to introduce both and, in a companion post, put one to work on a real puzzle.

Selection Sort

The idea: repeatedly select the smallest remaining value and place it in the next position — smallest into slot 0, next-smallest into slot 1, and so on until every element is placed.

  • Pseudocode
  • Python
SELECTION-SORT(a):
    n ← length(a)
    for i ← 0 to n-1:
        min ← i
        for j ← i+1 to n-1:
            if a[j] < a[min]:
                min ← j
        swap a[i] and a[min]      # at most n-1 swaps total
    return a
def selection_sort(a):
    n = len(a)
    for i in range(n):
        smallest = i
        for j in range(i + 1, n):
            if a[j] < a[smallest]:
                smallest = j
        a[i], a[smallest] = a[smallest], a[i]   # pythonic swap, no temp needed
    return a

original = [5, 3, 4, 1, 2, 10, 1]
print("input: ", original)
print("sorted:", selection_sort(original.copy()))   # .copy(): it sorts in place
input:  [5, 3, 4, 1, 2, 10, 1]
sorted: [1, 1, 2, 3, 4, 5, 10]

The distinctive property: selection sort makes at most n − 1 swaps, the fewest of the elementary sorts. That barely matters in RAM, but it matters a lot when a write is the expensive operation.

Insertion Sort

The idea is how most people sort a hand of cards. Keep the cards to your left sorted; pick up the next one; slide it left past every larger card until it lands in the right spot.

  • Pseudocode
  • Python
INSERTION-SORT(a):
    n ← length(a)
    for i ← 0 to n-1:
        for j ← i downto 1:
            if a[j] < a[j-1]:
                swap a[j] and a[j-1]
            else:
                break             # already in order from here; stop early
    return a
def insertion_sort(a):
    n = len(a)
    for i in range(n):
        for j in range(i, 0, -1):             # lazy range: the break actually saves work
            if a[j] < a[j-1]:
                a[j], a[j-1] = a[j-1], a[j]    # swap into place
            else:
                break                          # already in order here; stop early
    return a

original = [5, 3, 4, 1, 2, 10, 1]
print("input: ", original)
print("sorted:", insertion_sort(original.copy()))
input:  [5, 3, 4, 1, 2, 10, 1]
sorted: [1, 1, 2, 3, 4, 5, 10]

How they scale

Claims about running time are cheap; let’s measure. The two functions above are timed below on three kinds of input — randomly shuffled, already sorted, and reversed — across growing sizes, taking the best of a few trials at each size.

Show the benchmark code 
import time, random
import matplotlib.pyplot as plt

def best_ms(sort, data, trials=3):
    best = float("inf")
    for _ in range(trials):
        d = data.copy()                 # fresh copy: the sorts mutate in place
        t = time.perf_counter()
        sort(d)
        best = min(best, time.perf_counter() - t)
    return best * 1000

sizes = [100, 200, 400, 800, 1600]
cases = {
    "random":   lambda n: random.sample(range(n * 3), n),
    "sorted":   lambda n: list(range(n)),
    "reversed": lambda n: list(range(n, 0, -1)),
}
random.seed(0)

fig, axes = plt.subplots(1, 2, figsize=(9, 3.6), sharey=True)
for ax, (name, algo) in zip(axes, [("Selection sort", selection_sort),
                                   ("Insertion sort", insertion_sort)]):
    for label, gen in cases.items():
        ax.plot(sizes, [best_ms(algo, gen(n)) for n in sizes],
                marker="o", label=label)
    ax.set_title(name)
    ax.set_xlabel("input size n")
    ax.grid(alpha=0.3)
axes[0].set_ylabel("time (ms)")
axes[1].legend(title="input order")
plt.tight_layout()
plt.show()

Two things jump out. Selection sort’s three curves lie almost on top of one another — it doesn’t care whether the input is sorted, shuffled, or reversed, because it always scans the entire unsorted remainder and always makes the same n − 1 swaps. Input order is invisible to it.

Insertion sort fans out completely. On sorted input it is essentially flat: the break fires on the first comparison of every pass, the work is linear, and at n = 1600 it runs on the order of a hundred times faster than selection sort. On random input it is quadratic and slightly worse than selection (more data movement). On reversed input — its worst case, where the break never fires — it is the slowest line on the whole plot. The same algorithm is both the fastest and the slowest curve in the figure, decided entirely by the order of its input. That is what adaptive means, made visible.

Where these still earn their keep

  • Insertion sort is online. It can absorb elements one at a time as they arrive, keeping the array sorted after each insertion — so it fits streaming input, where a divide-and-conquer sort would have to wait for the whole batch.
  • Insertion sort is adaptive. On nearly-sorted input the inner loop hits the break almost immediately, so it approaches O(n). This is why real-world hybrid sorts (like the Timsort behind Python’s own sorted) fall back to insertion sort on short runs, where its low overhead beats an asymptotically faster algorithm.
  • Selection sort minimizes writes. Its n − 1 swaps make it attractive when writing is costly or wearing — the seed of the companion post’s puzzle.
